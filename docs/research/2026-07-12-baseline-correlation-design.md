# 생활인구 ↔ citydata 혼잡 상관 실험 설계 (베이스라인 선행 게이트)

- 상태: 설계 확정, 실행 대기 (데이터 겹침 창 미충족)
- 작성일: 2026-07-12
- 관련: [ADR-0009](../adr/ADR-0009-living-population-250m-grid.md),
  [생활인구 리서치](2026-07-12-living-population.md), Track 1

## 1. 왜 이 실험이 먼저인가

2층 엔진(베이스라인 × 실시간 이상치 전파)의 베이스라인 층으로 생활인구를 쓰려는
계획의 숨은 가정: **존재 인구(실내 포함)가 보행 혼잡과 충분히 같이 움직인다.**
생활인구 포털 정의는 명시적으로 존재 인구이지 보행 흐름이 아니므로, 이 상관은
가정이 아니라 검증 대상이다. 상관이 약하면 베이스라인 층 설계를 바꿔야 하고, 그걸
모른 채 파이프라인을 쌓으면 매몰 비용이 된다.

## 2. 설계

### 공간 정합

- 각 핫스팟 폴리곤(공식 121개)과 교차하는 250m 셀 집합을 구한다.
- 가중치 `wᵢ = area(cellᵢ ∩ hotspot) / area(cellᵢ)`. 즉 셀 인구 중 핫스팟
  내부에 속하는 비율을 면적 비례로 추정해 `SPOP_hotspot(t) = Σ wᵢ × SPOPᵢ(t)`로 합친다.
- 선행 조건: CELL_ID → geometry 변환 검증 완료.

### 신호 선택

- citydata 쪽은 `ppltn_min/max`의 중앙값(`ppltn_mid`)을 주 신호로 한다. 4단계
  레벨보다 정보량이 많고 이미 snapshot에 저장돼 있다. 레벨은 보조 참고.
- 생활인구 쪽은 셀별 `SPOP` 총계(파일 컬럼 `생활인구합계`). **2026-07-12 파일 실측으로
  총계도 `*` 마스킹될 수 있음을 확인**(심야 저인구 셀) — API 5행 표본의 "총계는 마스킹
  없음" 추정은 파일 전체에서는 성립하지 않는다.
- 마스킹 처리 규칙(관측 전 확정): `*` 총계는 2.0으로 대치(1~3 구간 중앙값)하고,
  0과 3 대치로 민감도 검사를 병행한다. 핫스팟 교차 셀 합계에서 마스킹 셀의 기여는
  셀당 ≤3명으로 유계이므로 결과를 뒤집기 어렵지만, 마스킹 비율은 항상 함께 보고한다.

### 시간 정합

- 생활인구 `YMD/TT`는 KST 기준으로 추정(`[VERIFY]` — 파일 명세로 확정).
- snapshot `observed_at`은 UTC 저장이므로 KST 변환 후 정시 매칭. 해당 정시
  ±5분 내 최신 snapshot을 사용하고, 없으면 그 시간 쌍은 결측 처리한다(보간 금지).

### 데이터 창

- 생활인구는 약 4일 지연 공개. 유효 표본은 **snapshot 수집이 연속인 날**로 한정한다.
- 현재 snapshot 이력은 1~2일 + 공백이라 실험 불가. **연속 수집 ≥7일이 선행 조건**
  — 전용 worker(ADR-0008)는 이 실험의 연구 인프라다.

### 지표 (관측 전 사전 정의)

1. 핫스팟별 시간별 Spearman(`SPOP_hotspot`, `ppltn_mid`) — 동시성 상관.
2. **요일×시간 평균 프로파일 간 상관** — 베이스라인 용도에 더 직접적인 주 지표.
   (베이스라인은 "평소 패턴"이므로 순간 상관보다 프로파일 상관이 판정 기준.)
3. 유효 쌍 수, 결측·마스킹 비율, 핫스팟 카테고리별(관광특구/발달상권/인구밀집) 분해.

### 판정 기준 (초기값, 데이터 관측 전 설정)

| 결과 | 기준 | 행동 |
|---|---|---|
| 사용 가능 | 프로파일 상관 중앙값 ≥ 0.6 이고 하위 10% 핫스팟 ≥ 0.3 | 베이스라인 층 구현 진행 |
| 조건부 | 중앙값 0.4~0.6 | 카테고리별 분해로 사용 범위 제한, 제한 사유 기록 |
| 불가 | 중앙값 < 0.4 | 생활인구 직접 사용 포기, 베이스라인 재설계 |

이 실험은 탐색적 연구이며 Phase 6 같은 확증 게이트가 아니다. 다만 기준을 관측 후에
바꾸면 사유를 명시하고 "탐색" 섹션으로 격리한다 — 사전 등록 문화(PHASE6_PREREGISTRATION)
와 같은 규율을 완화된 강도로 적용한다.

## 3. 실행 형태

- 오프라인 결정적 스크립트(가칭 `run_baseline_correlation.py`, 구현은 겹침 창 확보 후):
  입력 = 일별 grid 파일 + snapshot DB, 출력 = 핫스팟별 지표 표.
- 결과는 `docs/EVAL_REPORT.md` 부록 또는 `docs/research/`에 기록하고, 판정과 행동을
  ADR로 남긴다.

## 4. 순서 의존성

1. `[HUMAN]` 전용 worker로 snapshot 연속 수집 시작 (ADR-0008)
2. 일별 grid 파일 확보 루틴 (다운로더 구현·일별 파일 실검증 완료)
3. CELL_ID geometry 검증 (표본 검증 완료, 공식 경계 전수 대조 `[VERIFY]`)
4. 겹침 ≥7일 축적 후 실험 실행

1이 시작되지 않으면 2~3이 끝나도 실험은 열리지 않는다. 현재 전체 경로의 병목은
여전히 worker다.
