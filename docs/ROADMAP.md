# cafe-crowd 병렬 고도화 로드맵

이 문서는 서울 MVP 이후의 작업을 세 개 제품 트랙과 하나의 공통 플랫폼 레인으로
분리한다. 세부 계획은 각 트랙 문서가 담당하고, 충돌하는 경우 `docs/PLAN.md`와 채택된
ADR이 우선한다.

## 2026-07-12 외부감사 우선순위 잠금

핵심 가설의 현장 검증과 실시간 production이 완료될 때까지 Track 2·3의 신규 구현과 신규
확장 문서 작성을 동결한다. 기존 문서와 계약은 보존하되 확장하지 않는다.
사용자의 2026-07-13 요청에 따른 read-only feasibility 조사는 예외로 수행했지만, 런타임
확장 동결은 유지한다. ADR-0010에 따라 두 번째 provider fixture 전 범용 계약 확장도 금지한다.
관리형 PostgreSQL read 전환은 완료됐다. 현재 제품 우선순위는 Track 1의 polygon shadow와
Confidence V2 검증, 축소 Phase 6 현장 관측이다. 전용 상시 worker 안정화와 백업·복구는
운영 레인에서 병렬 진행하되 엔진 실측을 대체하지 않는다. 근거와 예외는
`audits/2026-07-12-external-review-response.md`, `SHADOW_REPORT.md`와 ADR-0008에 기록한다.

## 운영 원칙

- 트랙 1은 모델 품질과 신뢰도 의미를 소유한다.
- ADR-0011에 따라 도시 활동도 surface가 코어이며, 카페는 첫 overlay다.
- 트랙 2와 트랙 3은 지역별 데이터 소스 검증과 확장을 소유한다.
- 지도, 저장소, 타일 API, 관측성은 공통 플랫폼으로 한 번만 구현한다.
- 카페 위치는 전 세계로 확장할 수 있지만, 혼잡도는 검증된 데이터가 있는 영역에서만
  표시한다.
- 확인되지 않은 외부 데이터 소스, 라이선스, 갱신 주기는 `[VERIFY]`로 유지한다.

## 공통 플랫폼 레인

### 실시간 운영

- 관리형 PostgreSQL을 정본으로 사용하고 API와 10분 ingest worker를 분리한다.
- 공급자별 수집기는 공통 observation 계약으로 정규화한다.
- stale, 실패율, cycle duration, 마지막 성공 시각을 도시와 공급자 단위로 기록한다.

### 지도 성능

- 저배율은 서버 집계 셀, 고배율은 최소 필드 벡터 타일을 제공한다.
- 지도 타일에는 위치, ID, 혼잡도, 신뢰도, 커버리지만 포함한다.
- 이름, 주소, 연락처, 근거와 추이는 선택된 장소 상세 API에서만 제공한다.
- snapshot/model/catalog version을 캐시 키와 ETag에 포함한다.
- 목표는 지도 데이터 p95 150ms 이하, 캐시 hit p95 50ms 이하, 일반 화면 payload
  200KB 이하, 모바일 지도 상호작용 가능 시점 3초 이하다.

### 유니버설 데이터 경계(실험적)

ADR-0010에 따라 아래 항목은 두 번째 provider fixture 전까지 확정 런타임 계약이 아니라
seam inventory다. 실측에서 공통으로 확인된 최소 필드만 승격한다.

- `PlaceCatalogProvider`: 장소 원장과 안정적인 장소 ID를 제공한다.
- `CrowdObservationProvider`: 관측 영역, 원본 값, 시각, 갱신 SLA와 출처를 제공한다.
- `CoverageProvider`: 지원 영역, 해상도, 현행성과 라이선스 상태를 제공한다.
- `RegionProfile`: 지역, 국가, 시간대, 언어, 단위와 레벨 정규화 정책을 제공한다.
- 도시별 원본 레벨은 지역 내 상대 기준으로 정규화한다. 서로 다른 도시의 원본 숫자를
  직접 비교하지 않는다.

### 지도 제품 모드

1. **도시 활동도**: 검증된 폴리곤·격자에 자기 평소 대비 지역 활동도를 표시한다.
2. **카페 찾기**: activity surface 위에 카페를 첫 overlay로 표시한다. 매장 좌석 점유율로
   표현하지 않는다.
3. **데이터 상태·커버리지**: observed/forecast/baseline/stale/unsupported와 지원 경계를
   표시한다.

## 의존성과 병렬화

- 세 트랙의 조사, fixture 확보, 평가 프로토콜 작성은 병렬로 진행한다.
- 두 번째 provider fixture 전에는 observation/coverage 계약을 확장하지 않는다.
- 트랙 2는 국내에서 유니버설 계약을 검증하는 첫 확장 사례다.
- 트랙 3은 데이터 라이선스 검증을 통과하기 전 제품 구현을 시작하지 않는다.
- 트랙 1의 confidence V2는 서울 평가를 통과한 후 다른 도시에 기본값으로 배포한다.
  도시별 보정값은 각 도시 평가셋을 통과해야 활성화한다.

## 단계별 릴리스 게이트

### M1 — 서울 기반선

- 실시간 PostgreSQL 운영, 벡터 타일, 관측 지표를 연결한다.
- 서울 밀집도·카페·커버리지 모드를 제공한다.
- 현행 모델과 confidence V2를 같은 평가셋에서 비교한다.

### M2 — 국내 확장

- 서울 외 후보 도시의 데이터 현행성, 공간 해상도, 재배포 라이선스를 검증한다.
- 첫 도시를 유니버설 어댑터로 추가하고 서울 코드의 조건 분기를 금지한다.
- 미지원 지역은 POI만 표시하고 혼잡도는 표시하지 않는다.

### M3 — 해외 확장

- 후보 도시별 데이터 소스와 비용을 비교한다.
- 시간대, 다국어, 단위, attribution, 개인정보 요구사항을 검증한다.
- 한 도시의 제한 공개 후 정확도와 비용을 통과해야 다음 도시를 추가한다.

## 중단 조건

- 원본 데이터의 상업적 재배포 권한이 불명확한 경우
- 실시간을 표방하지만 실제 갱신 지연을 측정할 수 없는 경우
- 카페 주변 혼잡도를 매장 점유율로 오인하게 만드는 경우
- 평가셋에서 기존 서울 모델보다 성능이 낮고 원인을 설명하지 못하는 경우
- 공급자 비용이 도시별 사용량 예산을 초과하는 경우
